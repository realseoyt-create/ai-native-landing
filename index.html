1단계 · AI Transformation

지금 하고 있는 것

4가지 과제를 동시에 진행하며 AI Native를 위한 역량과 자산을 쌓고 있습니다.

AXIS 01 · AI NATIVE 전환

하네스 배포 · 팀원 자율 개발

팀원 스스로 AI를 활용해서 일하게

환경 설정 1회로 즉시 시작 가능한 하네스 플러그인 배포. 오픈코드 기반으로 보안 이슈 없이 사내에서 운용하며, 각자의 업무에 맞는 페르소나 에이전트를 자율 개발합니다.

  • ▸오픈코드 기반, 보안 이슈 제거
  • ▸환경 설정 1회 → 즉시 사용
  • ▸페르소나 에이전트 자율 개발
  • ▸DevOps 적용 — 클라우드 서버 안전 배포
AXIS 02 · FDE 과제 밀착

현장 AI 전환 사례 발굴

과제에 밀착해서 처음부터 끝까지 함께

TSS 엔지니어가 과제별로 밀착 투입됩니다. 요청 대기가 아니라 함께 몰입해서 반복 업무를 자동화하고, 성공 사례를 재현 가능한 하네스로 패키징해 팀 전체로 확산합니다.

  • ▸과제 선정 → TSS 밀착 투입 → 함께 완결
  • ▸Excel 업무 제거 · E2E Workflow 개선
  • ▸성공 사례 → 하네스 패키징 → 팀 확산
  • ▸MBO 연계 AI 전환율 커버리지
AXIS 03 · TSP LLM 통합 Agent

파편화된 지식을 하나로

팀의 노하우를 LLM 플랫폼에 통합

각자의 머릿속, 로컬 파일, 메신저에 흩어진 지식을 TSP LLM 플랫폼 위에 통합합니다. Test기술팀 전용 Agent가 누구에게나 팀의 노하우를 전달합니다.

  • ▸TSP LLM 플랫폼 기반 팀 전용 Agent 구축
  • ▸파편화 지식 → 구조화된 공유 자산
  • ▸신규 팀원 온보딩 시간 단축
  • ▸센터 연합 과제 (3차 회의 진행 중)
AXIS 04 · AI Ready Data

Machine Readable 데이터 확보

AI가 실제로 쓸 수 있는 데이터 기반

TSP Data Lake 기반으로 정형·비정형 데이터를 분류·정비합니다. 메타데이터를 체계화해 AI가 데이터를 신뢰하고 활용할 수 있는 Reliable Data로 전환합니다.

  • ▸데이터 분류 체계 수립 · 메타데이터 정비
  • ▸정형·비정형 적재 목표 50%
  • ▸Data Curation → Machine readable
  • ▸MBO 연계
AXIS 01 vs 02 · 두 가지 실행 방식

어떻게 다른가

AI 전환은 "팀원 스스로 영리해지는 것"과 "과제에 밀착해서 함께 바꾸는 것", 두 방향으로 동시에 이루어집니다. 둘은 보완적이지 배타적이지 않습니다.

AXIS 01 · AI NATIVE 🧑‍💻

팀원 스스로 AI를 활용해서

환경 설정 한 번으로 AI 작업 환경을 즉시 갖춥니다. 오픈코드 기반으로 보안 이슈 없이 사내에서 바로 사용 가능하며, 각자의 업무에 맞는 에이전트를 직접 만들 수 있습니다.

  • 👥대상: Test기술팀 전 팀원
  • 🎯목표: 팀원 AI 활용 역량 강화, 업무 효율 향상
  • 🛠️방식: 오픈코드 하네스 배포, 에이전트 자율 개발
  • 📦결과물: 페르소나 에이전트, 팀 하네스 기여
AXIS 02 · FDE 🤝

과제에 밀착해서 함께

TSS 엔지니어가 과제별로 밀착 투입됩니다. 멀리서 지원하는 게 아니라 처음부터 끝까지 함께 몰입해 실제 Pain Point를 해결하고, 성공 사례를 팀 전체로 확산합니다.

  • 🎯대상: Test기술팀 내 각 과제 (TSS 밀착 투입)
  • 💡목표: 과제 완결, 반복 업무 자동화, 성공 사례 축적
  • 🛠️방식: 함께 몰입, 처음부터 끝까지 책임지고 구현
  • 📦결과물: 재현 가능한 하네스 패키지, 팀 전체 확산

2단계 · AI Native (목표)

일하는 방식의 대전환

AI Transformation이 쌓은 자산 위에서, 팀원 모두가 Agent를 통해 일합니다. 도구도, 데이터도, 배포도 — 모두 신뢰할 수 있는 상태가 됩니다.

AI NATIVE · 팀원의 일하는 방식
팀원
⇄
Agent
⇄
Job

팀원이 직접 실행하던 업무를 Agent가 처리합니다.
팀원은 방향을 설정하고, Agent가 실행하고, Job이 완결됩니다.

RELIABLE DATA

AI와 사람이 믿고 쓰는 데이터

AI Ready Data 과제가 쌓은 자산이 Reliable Data가 됩니다. Agent가 판단의 근거로 사용하고, 사람이 결과를 신뢰할 수 있는 데이터 환경입니다.

Reliable Harness & APIs

누구나 믿고 쓸 수 있는 하네스와 API 레이어. 팀이 만든 에이전트들이 이 위에서 안정적으로 동작합니다. 개인 개발 → 팀 표준 → 조직 인프라로 발전합니다.

Test기술팀 단위 확산

서버 접속 없는 배포 체계

AI Native가 실현되었을 때, 배포는 더 이상 서버에 직접 접속하거나 운영에 영향을 주지 않습니다. DevOps 기반이 완성되어 Agent가 배포 파이프라인을 관리합니다.

DevOps 완성 목표
공통 기반 · DevOps

서버 접속 없는 안전한 배포 체계

AI Native의 핵심 전제 중 하나입니다. Agent가 배포를 실행할 수 있으려면, 배포 자체가 안전하고 자동화되어 있어야 합니다.

🔄

배포 패러다임의 전환

사람이 서버에 직접 접속해 배포하던 방식에서, 검증된 파이프라인이 안전하게 자동 배포하는 방식으로

기존 방식
  • ⛔서버 직접 SSH 접속 후 수동 배포
  • ⛔담당자마다 다른 배포 절차
  • ⛔배포 실패 시 수동 롤백, 긴 복구 시간
  • ⛔운영 중 영향 여부 사전 확인 불가
AI Native 목표 상태
  • ✅서버 접속 없이 파이프라인으로 배포
  • ✅표준화된 절차, 누가 해도 동일한 결과
  • ✅이상 감지 시 자동 롤백, 즉각 복구
  • ✅배포 전 사전 검증으로 운영 영향 Zero
🏗️

실행 환경 일치

개발·스테이징·운영 환경을 동일하게 유지합니다. "내 환경에서는 됐는데" 문제를 구조적으로 제거합니다.

⚡

자동 빌드 · 배포

코드 변경부터 서버 반영까지 파이프라인이 자동으로 처리합니다. 사람의 개입 없이 검증된 배포가 이루어집니다.

🛡️

실시간 이상 감지 · 자동 롤백

배포 후 실시간으로 이상을 감지하고, 문제 발생 시 자동으로 이전 상태로 복원합니다. AI가 이 판단을 지원합니다.

Reliable Harness · 핵심 구성

하네스(Harness)란

AI가 일을 잘 할 수 있도록 설계된 작업 환경 패키지입니다. 오픈코드 기반으로 구성해 보안 이슈 없이 사내에서 운용하며, 환경 설정 한 번으로 팀원 누구나 즉시 사용할 수 있습니다.

harness / 6가지 구성 레이어
LAYER 01

컨텍스트

AI가 팀의 규칙과 프로젝트를 이해하도록 설계된 지식 레이어.
CLAUDE.md

LAYER 02

스킬(Skills)

반복 워크플로우를 명령어 하나로 실행하는 재사용 가능한 레시피.
/skill-name

LAYER 03

훅(Hooks)

파일 수정, 명령 실행, 완료 시점에 자동으로 반응하는 트리거.
PreToolUse / PostToolUse

LAYER 04

메모리(Memory)

대화가 끊겨도 팀과 프로젝트를 기억하는 영속 기억 레이어.
memory/*.md

LAYER 05

도구 연동(MCP)

슬랙, 텔레그램, DB, GitHub 등 외부 도구를 AI가 직접 다루도록 연결.
MCP Servers

LAYER 06

에이전트 오케스트레이션

복잡한 작업을 여러 에이전트가 분담해 병렬로 처리하는 구조.
fork / subagent
